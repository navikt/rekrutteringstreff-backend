# Alternativ 2.5: PostgreSQL med view (rekrutteringstreffsøk)

## Idé

Søk mot et database-view som samler all relevant data fra eksisterende tabeller i en flat struktur — uten egen søketabell, uten synkroniseringslogikk, uten indekser.

Dette er en mellomting mellom alternativ 2 (søketabell) og alternativ 3 (direkte queries med joins). Vi får den flate, lettleste strukturen fra alternativ 2, men slipper å holde en egen tabell oppdatert. Og vi slipper å skrive joins i hver søke-query som i alternativ 3.

---

## Hvorfor dette alternativet

- View gir en denormalisert struktur uten synkroniseringskode
- Alltid konsistent — viewet leser direkte fra kildetabellene
- Ingen migrasjoner utover å opprette viewet
- Ingen reindeksering — gjenskaping av viewet er en `CREATE OR REPLACE VIEW`
- Enklere queries i repository-laget enn med joins
- Lett å utvide med flere kolonner og filtre
- Naturlig migrasjonssti til søketabell (alternativ 2) hvis vi senere trenger indekser

---

## Scope: minimal versjon

Minimal versjon inkluderer eier, status og Nav-kontor:

**Filtre:**

- **Eierfilter** (tabs): Alle / Mine / Mitt kontor
- **Statusfilter**: Utkast, Publisert, Søknadsfrist passert, Fullført, Avlyst
- **Kontorfilter**: Velg ett eller flere Nav-kontorer (kun synlig på «Alle»-taben)

`kontorer` lagres allerede som `text[]` med enhetId-er (f.eks. `'0318'`) på `rekrutteringstreff`-tabellen. Frontend har en komplett mapping mellom enhetId og kontornavn i `enheter.json`, som også fungerer som listen over alle Nav-kontorer. Backend trenger derfor ikke returnere label for kontorer i aggregeringen — kun enhetId og antall.

**Hva dette gir:**

- Brukerne kan finne treff de er eier/medeier av
- Brukerne kan filtrere på status og Nav-kontor
- Antall per filterverdi i filterpanelet (aggregeringer for status og kontor)
- Paginering (25 per side)

**Hva dette ikke gir:**

- Fritekst (søk på tittel, arbeidsgivernavn osv.)
- Geografifiltrering (fylke/kommune)
- Sorteringsvalg utover default
- Typo-toleranse eller stemming

## View-definisjon

```sql
CREATE OR REPLACE VIEW rekrutteringstreff_sok_view AS
SELECT
    rt.id,
    rt.tittel,
    rt.beskrivelse,
    rt.status,
    rt.fratid AS fra_tid,
    rt.tiltid AS til_tid,
    rt.svarfrist,
    rt.gateadresse,
    rt.postnummer,
    rt.poststed,
    rt.kommunenummer,
    rt.kommune,
    rt.fylkesnummer,
    rt.fylke,
    rt.opprettet_av_tidspunkt,
    rt.sist_endret,
    rt.eiere,
    rt.kontorer,
    CASE
        WHEN rt.status = 'UTKAST' THEN 'UTKAST'
        WHEN rt.status = 'AVLYST' THEN 'AVLYST'
        WHEN rt.status = 'FULLFØRT' THEN 'FULLFORT'
        WHEN rt.status = 'PUBLISERT' AND rt.svarfrist IS NOT NULL AND rt.svarfrist < now() THEN 'SOKNADSFRIST_PASSERT'
        WHEN rt.status = 'PUBLISERT' THEN 'PUBLISERT'
    END AS visningsstatus
FROM rekrutteringstreff rt
WHERE rt.status != 'SLETTET';
```

Viewet inneholder bare kolonner fra `rekrutteringstreff`-tabellen. Det kan utvides med subqueries eller joins mot `arbeidsgiver` og `innlegg` senere, men i denne versjonen trenger vi det ikke.

`visningsstatus` beregnes i viewet slik at all kode som bruker viewet slipper å gjenta CASE-uttrykket.

## Vanlig view vs. materialized view

Vi bruker et vanlig `VIEW`, ikke `MATERIALIZED VIEW`.

| Egenskap    | VIEW                        | MATERIALIZED VIEW           |
| ----------- | --------------------------- | --------------------------- |
| Konsistens  | Alltid oppdatert            | Stale til neste `REFRESH`   |
| Ytelse      | Evalueres ved query-tid     | Forhåndsberegnet            |
| Indekser    | Nei (bruker kildetabellens) | Ja, kan ha egne indekser    |
| Refresh     | Ikke nødvendig              | `REFRESH MATERIALIZED VIEW` |
| Vedlikehold | Null                        | Trenger refresh-strategi    |

Med forventet volum (sjelden over 1 000 aktive treff, maks ~100 000 totalt) er et vanlig view tilstrekkelig. PostgreSQL optimerer queries mot views på samme måte som vanlige queries — viewet er bare en navngitt query som inlines av planner.

Hvis ytelsen viser seg å bli et problem, kan vi:

1. Bytte til `MATERIALIZED VIEW` med periodisk refresh
2. Legge indekser på den materialiserte viewen
3. Gå videre til alternativ 2 (søketabell) med full kontroll

## Eksempel-queries

### Treffliste med eier + status + kontor

```sql
SELECT id, tittel, visningsstatus, fra_tid, til_tid, svarfrist,
       gateadresse, postnummer, poststed,
       kommune, fylke, opprettet_av_tidspunkt, sist_endret,
       eiere, kontorer
FROM rekrutteringstreff_sok_view
WHERE lower(:navident) = ANY(eiere)                     -- eierfilter
  AND visningsstatus = ANY(:visningsstatuser)            -- statusfilter
  AND kontorer && :kontorer::text[]                      -- kontorfilter (overlap)
ORDER BY sist_endret DESC
LIMIT :antallPerSide OFFSET :side * :antallPerSide;
```

### Statusaggregering

Uten statusfilter, med eier- og kontorfiltre:

```sql
SELECT visningsstatus, count(*) AS antall
FROM rekrutteringstreff_sok_view
WHERE lower(:navident) = ANY(eiere)
  AND kontorer && :kontorer::text[]
GROUP BY visningsstatus
ORDER BY visningsstatus;
```

### Kontoraggregering

Uten kontorfilter, med eier- og statusfiltre. Siden `kontorer` er et array, må vi bruke `unnest` for å telle per kontor:

```sql
SELECT kontor AS verdi, count(*) AS antall
FROM rekrutteringstreff_sok_view, unnest(kontorer) AS kontor
WHERE lower(:navident) = ANY(eiere)
  AND visningsstatus = ANY(:visningsstatuser)
GROUP BY kontor
ORDER BY antall DESC;
```

Label (kontornavn) slås opp i frontend via `enheter.json` — backend returnerer kun enhetId og antall.

## Styrker

- **Ingen synkronisering.** View leser alltid fra kildetabellene — ingen risiko for stale data
- **Enklere queries.** Repository-koden leser fra en flat view istedenfor å gjenta joins
- **Enkel å utvide.** Nye kolonner og filtre kan legges til med `CREATE OR REPLACE VIEW`
- **Ingen reindeksering.** Gjenskaping av viewet er en DDL-operasjon, ingen data å flytte
- **Naturlig migrasjonssti.** Kan bytte til søketabell (alternativ 2) eller OpenSearch (alternativ 1) uten å endre API-kontrakt eller frontend

## Svakheter

- **Ingen egne indekser** (med vanlig view). Bruker indekser fra kildetabellene
- **Ingen fritekst.** View alene løser ikke fulltekstsøk — det krever tsvector eller søketabell
- **Beregnet visningsstatus.** Planner kan ikke bruke indeks direkte på CASE-uttrykket. For filtrering på visningsstatus må den dynamiske SQL-en oversette til predikat på `status` + `svarfrist`, eller vi aksepterer seq scan (greit med <10k aktive rader)
- **Ingen ytelsesgevinst over joins.** Et vanlig view er bare en navngitt query som PostgreSQL inliner — gevinsten er lesbarhet i koden, ikke ytelse
- **Flere queries per request.** Med aggregeringer for status og kontor kjøres potensielt 3 queries per søke-request. Uproblematisk ved lavt volum, men bør måles tidlig
- **Skaleringstak** likt alternativ 3 — ved svært mange rader og komplekse filtre kan ytelsen falle

## Når vi trenger mer

Alternativ 2.5 er ment som et pragmatisk utgangspunkt. Når ett av disse behovene melder seg, er det naturlig å gå videre:

| Behov                                            | Neste steg                                     |
| ------------------------------------------------ | ---------------------------------------------- |
| Fritekst (tittel, beskrivelse, arbeidsgivernavn) | Alternativ 2 (søketabell med tsvector/pg_trgm) |
| Egne indekser for søk                            | Materialized view eller alternativ 2           |
| Typo-toleranse / stemming                        | Alternativ 2 (pg_trgm) eller alternativ 1      |
| Volum over ~10 000 aktive                        | Alternativ 2 med indekser                      |
| Søkekvalitet på nivå med kandidatsøk             | Alternativ 1 (OpenSearch)                      |

## Logging og overvåking av søke-queries

Siden vi bevisst starter uten indekser og med et vanlig view, er det viktig å ha innsikt i hvordan queryene faktisk oppfører seg i produksjon.

**Responstid-logging:** Logg responstid for søke-endepunktet (hele requesten inkl. alle queries). Bruk standardmetrikkene i Javalin/Prometheus. Dette gir oss data til å vurdere om og når vi trenger å optimalisere.

**Alerts ved trege queries:** Sett opp alert hvis søke-endepunktet konsekvent svarer over en terskel (f.eks. p95 > 500ms). Dette er signalet på at vi bør vurdere indekser, materialized view eller overgang til alternativ 2.

**Per-query logging i repository:** Logg utføringstid for de individuelle queryene (treffliste, statusaggregering, kontoraggregering) slik at vi kan identifisere hvilken query som eventuelt blir flaskehalsen.

**EXPLAIN ANALYZE ved behov:** Ikke i produksjon, men kjør `EXPLAIN ANALYZE` på representative søkekombinasjoner i testmiljø for å verifisere at planner velger fornuftige planer. Spesielt viktig å sjekke:

- Filtrering på beregnet `visningsstatus` (seq scan forventet, men bør verifiseres)
- Aggregeringer med flere samtidige filtre
- Paginering med stor offset

Dette gir oss et faktabasert grunnlag for å avgjøre om vi trenger å gå videre til alternativ 2 — i stedet for å gjette.
